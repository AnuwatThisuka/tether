# tether Execution Plans (ExecPlans)

> **DO NOT EDIT THIS FILE**
> This file is the ExecPlan template and guide only.
> Create plans in the plans directory:
>
> - **All work**: `plans/<timestamp>-<plan-name>.md`
>
> Use `<timestamp>` in `YYYYMMDD-HHmm` format (e.g., `20260719-1045-snapshot-handoff.md`). This sorts plans most-recent-first. Create the `plans/` directory if it does not yet exist.

This document describes the requirements for an execution plan ("ExecPlan"), a design document that a coding agent can follow to deliver a working feature or system change to `tether`. Treat the reader as a complete beginner to this repository: they have only the current working tree and the single ExecPlan file you provide. There is no memory of prior plans and no external context.

`tether` is a Postgres sync engine distributed as a Go library. It embeds into an existing Go HTTP server — it is not a standalone service. It reads the Postgres WAL via logical replication, filters changes into per-client **shapes**, streams them over WebSocket, and accepts writes back as **mutations** applied by user-supplied handlers. This is a correctness-critical data system: silent data loss is worse than a crash. Agent rules live in `AGENTS.md`. The master design and phased build order live in `IMPLEMENT_PLAN.md` — ExecPlans must stay consistent with both.

## Invariants — a plan that breaks one is wrong

Before drafting, read the "Invariants" section of `AGENTS.md` (root). These are correctness and security rules. A plan whose implementation would break one is wrong even if every test passes. Restate the relevant invariant in your plan and explain how the design honors it. The rules, in short:

1. **Never acknowledge an LSN before the change is durably recorded.** Persist first, then `SendStandbyStatusUpdate`. Advancing the confirmed LSN ahead of persisted state means data is gone after a crash.
2. **Shape filters are resolved server-side from auth claims. Always.** A predicate must never originate from client input — that is a cross-tenant data leak.
3. **Mutations must be idempotent by key.** Dedupe inside the same transaction that applies the effect — never in a separate one. Clients replay after reconnect.
4. **Snapshot and stream must not overlap or gap.** Snapshot at LSN _N_, then deltas strictly after _N_. Off-by-one silently duplicates or drops rows.
5. **Schema drift halts the shape.** On a column set that disagrees with the cached relation, stop and surface an error. Never guess a mapping.
6. **Slot lag is bounded.** Never hold a replication slot open indefinitely. Prefer dropping the slot and forcing a re-snapshot over unbounded WAL growth.
7. **One slow client must not stall the others.** Fan-out is per-client buffered. Full buffer → disconnect that client; never block the WAL reader on a socket write.

If a task seems to require breaking one of these, the plan must stop and surface the conflict rather than work around it.

Also respect **scope** from `AGENTS.md`: do not plan CRDTs/auto-merge, joins in shapes, presence/cursors, multi-region coordination, non-Postgres databases, or an admin UI. If the task appears to need one, stop and re-scope.

## Process

Steps:

1. Discovery & Orientation: map the repo, name the scope, enumerate unknowns. Capture initial Assumptions and Open Questions in the ExecPlan.
2. Question-driven Clarification: ask focused, acceptance-oriented questions grouped by plan section. Maintain the Open Questions list and pre-link each item to a Decision Log placeholder.
3. Draft the Plan: complete the ExecPlan skeleton end-to-end (Purpose, Context, Plan of Work, Validation, Idempotence, etc.), calling out risks, dependencies, and which Invariants the work touches.
4. Resolve Questions: as answers arrive, immediately update the ExecPlan — move items from Open Questions to the Decision Log with rationale; adjust Plan of Work and Acceptance accordingly.
5. Approval Gate: present the updated ExecPlan for approval. Do not implement until approved.
6. Implementation & Validation: implement per the plan, update Progress with timestamps, and validate via `make test` / `make test-integration` / `make lint` as appropriate. Log learnings in Surprises & Discoveries.
7. Closeout: write Outcomes & Retrospective; ensure the plan remains self-contained and accurate.
8. Write your plan to `plans/<timestamp>-<plan-name>.md`.
9. Plan Lifecycle: when the plan is complete and a PR is created, move it to `plans/done/`. If abandoned, move it to `plans/abandoned/`.

Example questions:

I reviewed `internal/wal/` and `transport/` to understand the replication and resume path.

    Where should this new concern live?
    a) internal/wal/ (slot lifecycle, LSN checkpointing, pgoutput decoding)
    b) internal/shape/ (registry, predicates, per-shape log)
    c) transport/ (WebSocket server and resume logic)
    d) Root package / tether.go (public API — needs a changelog entry)
    e) Other (specify)

    How should we prove snapshot/stream handoff correctness?
    a) Integration test asserting exact row-set equality against the source table
    b) Unit test with synthetic WAL messages only (usually insufficient for WAL/offset work)
    c) Other (specify)

## How to use ExecPlans

When authoring an ExecPlan, follow this guide to the letter. Be thorough in reading (and re-reading) source material — start from the skeleton and flesh it out as you research.

When implementing an ExecPlan, do not prompt the user for "next steps"; proceed to the next milestone. Keep all sections up to date, add or split entries in the Progress list at every stopping point to state progress made and next steps. Resolve ambiguities autonomously (within the Invariants), and commit frequently.

When discussing an ExecPlan, record decisions in the Decision Log for posterity; it must be unambiguously clear why any change to the specification was made. ExecPlans are living documents — it must always be possible to restart from _only_ the ExecPlan and no other work.

## Design Standards

Before drafting, review `AGENTS.md` (root) to ensure your design aligns with established patterns.

**When planning, ask yourself:**

1. Are there similar implementations in the codebase to reference?
2. Does this approach follow existing architectural patterns (public API in root package only; library code not under `cmd/`)?
3. If deviating from conventions, is the rationale documented in the Decision Log?
4. Which Invariants does this work touch, and how does the design honor each?
5. Does this change touch WAL decoding, offsets, or snapshot handoff? If so, the plan **must** include an integration test against real Postgres — unit tests with fake WAL messages are insufficient.
6. Would this make the `internal/e2e` convergence test flaky? If so, the design is wrong — do not plan retries to silence flakiness.

## Requirements

NON-NEGOTIABLE REQUIREMENTS:

- Every ExecPlan must be fully self-contained: in its current form it contains all knowledge and instructions a novice needs to succeed.
- Every ExecPlan is a living document. Revise it as progress is made, as discoveries occur, and as decisions are finalized. Each revision stays fully self-contained.
- Every ExecPlan must enable a complete novice to implement the feature end-to-end without prior knowledge of this repo.
- Every ExecPlan must produce demonstrably working behavior, not merely code changes that "meet a definition".
- Every ExecPlan must define every term of art in plain language or not use it.
- Every ExecPlan must reference `AGENTS.md` conventions and name the Invariants the work touches.

Purpose and intent come first. Begin by explaining, in a few sentences, why the work matters from a user's perspective: what someone can do after this change that they could not before, and how to see it working. Then guide the reader through the exact steps: what to edit, what to run, what they should observe.

The agent executing your plan can list files, read files, search, build, and run tests. It knows no prior context and cannot infer what you meant from earlier milestones. Repeat any assumption you rely on. Do not point to external blogs or docs; if knowledge is required, embed it in the plan in your own words.

## tether-Specific Conventions

### Tech Stack Reference

- **Language**: Go (target the version in `go.mod`)
- **Distribution**: Go library embedding into the host HTTP server — not a standalone service
- **Database**: Postgres with `wal_level = logical`; replication via `jackc/pgx` + `jackc/pglogrepl`
- **Transport**: WebSocket via `coder/websocket`
- **Logging**: `log/slog` only (no `fmt.Println` outside `cmd/`)
- **Formatting**: `gofumpt` (no custom formatter)
- **Dependencies**: keep the direct dep list short. Current allow-list without asking: `jackc/pgx`, `jackc/pglogrepl`, `coder/websocket`. Every other new dependency needs explicit user approval first.

### Common Commands

    make test                # unit tests, no database required
    make test-integration    # requires Postgres with wal_level=logical
    make lint                # golangci-lint
    make fmt                 # gofumpt

Integration tests against real Postgres:

    make db-up               # docker compose, exposes :54321
    export TETHER_TEST_DSN='postgres://tether:tether@localhost:54321/tether?replication=database'
    make test-integration
    make db-down

If a needed Make target does not exist yet, add it to the Makefile rather than documenting a one-off invocation. Do not skip or delete integration tests to make a build pass — fix the code. Concurrency-sensitive code runs under `-race` in CI; do not disable it.

### Architecture Map

- `tether.go` — public API surface (compatibility commitment; changes need a changelog entry)
- `internal/wal/` — replication slot lifecycle, LSN checkpointing, pgoutput decoding
- `internal/shape/` — shape registry, predicate evaluation, per-shape log
- `internal/proto/` — wire format, framing, offset encoding
- `transport/` — WebSocket server and resume logic
- `cmd/` — dev tooling and test harnesses only, never library code
- `internal/e2e/` — convergence test (two offline clients, conflicting edits, cold restart, reconnect, identical final state)

Public API lives in the root package only. Anything under `internal/` can change freely.

### Domain Traps (encode these into plans that touch them)

- **`UPDATE` events may not contain every column.** Without `REPLICA IDENTITY FULL`, Postgres sends only changed columns. Assuming a complete row writes nulls over live data.
- **`TOAST`ed values arrive as unchanged-markers.** Treat as "not present", never as "empty".
- **Transaction boundaries matter.** Apply changes to the shape log atomically per source transaction. Emitting a partial transaction exposes clients to states that never existed.
- **A replication connection is not a normal connection.** It cannot run ordinary queries. Snapshotting needs a separate pool.
- **Reconnects are not rare.** Resume paths deserve proportionally more test coverage than the happy path.

### Code Patterns

- Wrap errors with `fmt.Errorf("...: %w", err)`. Preserve the chain.
- Exported identifiers get doc comments. Unexported ones only where the reasoning is non-obvious — comments explain _why_, not _what_.
- Prefer table-driven tests. Prefer real Postgres over fakes for WAL/offset/snapshot work.
- Do not put library code under `cmd/`.
- Root-package API changes need a changelog entry in the plan's Concrete Steps / Validation.

## Formatting

Format and envelope are simple and strict. Each ExecPlan is one markdown file. Show commands, transcripts, diffs, or code as indented blocks (4 spaces). Use two newlines after every heading, use `#`/`##`/etc., and correct list syntax.

Write in plain prose. Prefer sentences over lists. Avoid checklists, tables, and long enumerations unless brevity would obscure meaning. Checklists are permitted only in the `Progress` section, where they are mandatory. Narrative sections stay prose-first.

## Guidelines

Self-containment and plain language are paramount. If you introduce a phrase that is not ordinary English ("shape", "LSN", "pgoutput", "TOAST unchanged-marker", "replication slot"), define it immediately and remind the reader how it manifests in this repo (name the files or commands where it appears). Do not say "as defined previously" or "according to the architecture doc." Include the explanation here, even if you repeat yourself.

Anchor the plan with observable outcomes. State what the user can do after implementation, the commands to run, and the outputs they should see. Acceptance is behavior a human can verify ("after `make test-integration`, `TestSnapshotStreamHandoff` asserts exact row-set equality against the source table") rather than internal attributes ("added an LSN compare function"). If a change is internal, explain how its impact is still demonstrated — usually a test, a log line, or an integration transcript.

Specify repository context explicitly. Name files with full repository-relative paths, name functions and packages precisely, and describe where new files go. If touching multiple packages, include a short orientation paragraph explaining how those parts fit together.

Be idempotent and safe. Write steps so they can be run multiple times without damage or drift. If a step can fail halfway (a migration, a schema change, a slot create/drop), include how to retry or roll back.

Validation is not optional. Include instructions to run `make test`, and `make test-integration` when the change touches WAL, offsets, or snapshot handoff. Always include `make lint`. Describe expected outputs so a novice can tell success from failure. If Postgres is required, say so and give the `make db-up` / `TETHER_TEST_DSN` sequence.

## Common Failure Modes (Avoid These)

**Undefined jargon**: using terms like "shape", "LSN", "slot lag", or "TOAST" without defining them. If a novice wouldn't know the term, define it inline.

**Letter-of-the-law implementations**: code that technically meets the spec but does nothing meaningful. Example: "idempotent mutations" resulting in a dedupe check outside the apply transaction.

**Outsourcing key decisions**: saying "choose an appropriate library" or "handle errors as needed" instead of being prescriptive. Make these decisions explicit, and never add a third-party dependency without user approval.

**Assuming context**: "as we discussed", "the usual pattern", "following the existing approach" without spelling out what that means.

**Validation theater**: "verify it works" without the exact command, input, and expected output — especially claiming WAL/offset correctness from unit tests alone.

**Incomplete state transitions**: describing the happy path but not reconnect/resume, schema drift, slow-client disconnect, slot drop, or crash-before-ack.

**Silently breaking an Invariant**: any plan that acks an LSN before durable record, accepts a client-supplied predicate, non-idempotent mutation apply, snapshot/stream overlap or gap, continues after schema drift, leaves a slot open indefinitely, or blocks the WAL reader on a slow client. If the task appears to need this, surface the conflict.

**Scope creep into non-goals**: planning CRDTs, joins in shapes, presence, multi-region, or an admin UI.

## Milestones

When work spans multiple days or involves significant complexity, break it into milestones. Each milestone must be:

1. **Independently verifiable** — can be tested and validated on its own
2. **Incrementally valuable** — adds working functionality, not just scaffolding
3. **Narratively complete** — goal → work → result → proof

For each milestone, include a brief scope paragraph, what will exist at the end that did not before, the commands to run and acceptance criteria, and how to verify success before proceeding.

**Example milestone structure:**

    ### Milestone 1: Persist-before-ack for shape log appends

    Establishes that a decoded WAL change is written to the shape log before
    any standby status update advances the confirmed LSN. At completion, a
    crash between persist and ack replays the change; a crash after ack never
    loses it. No WebSocket fan-out changes yet.

    Scope:
    - Reorder persist and SendStandbyStatusUpdate in internal/wal
    - Add an integration test that kills mid-ack and asserts replay
    - Document the LSN ordering invariant in a short comment near the call sites

    Acceptance:
        make db-up
        export TETHER_TEST_DSN='postgres://tether:tether@localhost:54321/tether?replication=database'
        make test-integration
        # Expected: TestPersistBeforeAck_CrashReplay passes
        make lint

    Verify before proceeding: integration test passes, lint clean, no new dependency.

Progress tracks granular tasks; milestones tell the story. Both must exist for complex plans.

## Prototyping & Spike Milestones

When facing significant unknowns or risky technical decisions, include explicit prototyping milestones to de-risk before full implementation.

**When to use spikes:** validating pgoutput decode edge cases (partial UPDATE, TOAST markers); comparing two resume/offset encodings; measuring fan-out buffer pressure under a slow client; confirming snapshot LSN handoff semantics against real Postgres.

**Spike structure:**

    ### Spike: Snapshot LSN handoff has neither overlap nor gap

    Goal: Confirm a client that finishes a snapshot at LSN N receives stream
    deltas strictly after N, with exact row-set equality vs the source table.

    Approach:
    - Minimal integration harness under internal/e2e or the relevant package
    - Insert/update during snapshot window; assert final client state
    - Measure: duplicate keys and missing keys both zero

    Success criteria:
    - Exact row-set equality holds under concurrent writes during snapshot
    - OR: documented tradeoff / required API change for the Decision Log

    Outcome: [To be filled after spike completes]

**Guidelines for spikes:** keep them additive and isolated when possible; set a time box; document findings in Surprises & Discoveries; record the decision in the Decision Log with evidence. Prefer real Postgres over synthetic WAL fakes.

## Parallel Implementations & Safe Refactoring

When migrating or refactoring, prefer patterns that keep the codebase working throughout:

**Additive-then-subtractive**: add new code alongside old, verify tests pass with both, then remove the old. Never delete-then-replace in a single step.

**Parallel implementations**: during a wire-format or resume-path migration, keep both encodings or code paths temporarily. Validate equivalence with tests (especially reconnect/resume), then retire the old path.

**Feature flags**: for risky public-API changes, prefer an additive API plus deprecation over a silent behavior change. Root-package changes need a changelog entry.

Document parallel-implementation strategies in the Plan of Work and include cleanup steps. Never hold a replication slot "a bit longer" as a migration crutch (Invariant 6).

## Skeleton of a Good ExecPlan

    # <Short, action-oriented description>

    This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

    Reference: This plan follows conventions from AGENTS.md (root) and this template. Invariants touched: <list, or "none">.

    ## Purpose / Big Picture

    Explain in a few sentences what someone gains after this change and how they can see it working. State the user-visible (or embedder-visible) behavior you will enable.

    ## Assumptions

    State temporary assumptions that unblock planning. Every assumption must either be confirmed (moved to the Decision Log) or removed by implementation end.

    ## Open Questions

    List unresolved, acceptance-oriented questions. For each, note the impacted plan sections and add a placeholder in the Decision Log for the eventual answer.

    ## Progress

    Use a list with checkboxes to summarize granular steps. Every stopping point must be documented here, splitting a partially completed task into "done" vs. "remaining" as needed. This section must always reflect the actual current state.

    - [x] (2026-07-19 13:00Z) Example completed step.
    - [ ] Example incomplete step.
    - [ ] Example partially completed step (completed: X; remaining: Y).

    ## Surprises & Discoveries

    Document unexpected behaviors, bugs, optimizations, or insights. Provide concise evidence.

    - Observation: ...
      Evidence: ...

    ## Decision Log

    - Decision: ...
      Rationale: ...
      Date/Author: ...

    ## Outcomes & Retrospective

    Summarize outcomes, gaps, and lessons at major milestones or at completion. Compare the result against the original purpose.

    ## Context and Orientation

    Describe the current state as if the reader knows nothing. Name key files and packages by full path. Define any non-obvious term. Do not refer to prior plans.

    Always mention:
    - Which package(s) are affected (internal/wal, internal/shape, internal/proto, transport, root, cmd, internal/e2e)
    - Whether the change touches the public API in tether.go (changelog required if yes)
    - Whether it touches WAL decoding, LSN ack ordering, offsets, or snapshot/stream handoff (integration test required if yes)
    - Which Invariants are in play

    ## Plan of Work

    Describe, in prose, the sequence of edits and additions. For each edit, name the file and location (package, function) and what to insert or change. Keep it concrete and minimal.

    ## Concrete Steps

    State exact commands and where to run them (working directory is the repo root unless noted). Show a short expected transcript so the reader can compare.

        make test
        # Expected: ok  .../internal/shape

        make db-up
        export TETHER_TEST_DSN='postgres://tether:tether@localhost:54321/tether?replication=database'
        make test-integration
        # Expected: ok  .../internal/e2e

    ## Validation and Acceptance

    Describe how to exercise the system and what to observe.

    For unit-level changes:
        make test
        make lint
        # Package under test passes; new test asserts the behavior

    For changes needing real Postgres (WAL, offsets, snapshot handoff):
        make db-up
        export TETHER_TEST_DSN='postgres://tether:tether@localhost:54321/tether?replication=database'
        make test-integration
        make lint
        # Integration tests pass; handoff/equality assertions hold

    Always run before calling it done:
        make test
        make lint
        # Plus make test-integration when the change requires it

    If the change could affect reconnect or multi-client correctness, also ensure
    the convergence test in internal/e2e still passes without new retries.

    ## Idempotence and Recovery

    If steps can be repeated safely, say so. For slot create/drop, schema changes, or local docker DB state, provide a safe retry path (e.g. make db-down && make db-up).

    ## Artifacts and Notes

    Include the most important transcripts, diffs, or snippets as indented examples. Keep them concise and focused on what proves success.

    ## Interfaces and Dependencies

    Be prescriptive. Name the packages, types, and functions to use and why. Specify the Go types and function signatures that must exist at the end of the milestone. For a public API change, name the root-package symbols and note the changelog entry. For any new dependency beyond pgx/pglogrepl/websocket, stop and get user approval first — do not add it in the plan as a done deal.

If you follow the guidance above, a single, stateless agent — or a human novice — can read your ExecPlan top to bottom and produce a working, observable result that respects every Invariant. That is the bar: SELF-CONTAINED, SELF-SUFFICIENT, NOVICE-GUIDING, OUTCOME-FOCUSED, INVARIANT-SAFE.

When you revise a plan, ensure your changes are reflected across all sections, including the living-document sections, and write a note at the bottom describing the change and why. ExecPlans describe not just the what but the why.

## Plan Lifecycle

ExecPlans have a defined lifecycle that keeps the plans folder clean and provides a historical record.

### Directory Structure

    plans/
    ├── <active-plans>.md
    ├── done/
    └── abandoned/

### When to Move Plans

**To `done/`**: move the plan there when creating the PR that completes the work. Before moving, fill in the `Outcomes & Retrospective` section.

**To `abandoned/`**: move the plan there if work stops without completion. Add a note explaining why (scope changed, approach invalidated, deprioritized).

### Edge Cases

- **PR closed without merging**: the plan stays in `done/`. If work resumes, move it back to active and update `Progress`.
- **Plan spans multiple PRs**: keep it active until the final PR. Reference intermediate PRs in `Progress`, then move to `done/` on the final PR.
- **Reopening abandoned work**: move the plan from `abandoned/` back to active and update `Progress` to reflect the restart.
